Infrastructure recommendation for hosting the SnoCountry RAG server on an existing AWS account. This document evaluates EC2, Lightsail, Lambda, Fargate, and App Runner against the actual compute profile of the /api/rag/chat orchestrator defined in rag-plan.md.

1. Key insight — the server is an orchestrator

The RAG server is I/O-bound, not compute-bound. The heavy lifting — LLM inference, embeddings, and vector search — all lives off-box on Together.ai and Convex. The AWS server is a Node.js or Python process making HTTP calls and streaming SSE back. It needs almost no CPU or RAM.

Here is exactly what the /api/rag/chat action does on the server, step by step:

StepWhat happensResourceTime
1Validate + sanitize inputCPU (trivial)~0 ms
2Call Together.ai 8B for topic classificationOutbound HTTP~50 ms
3Regex-first intent classifierCPU (trivial)~0 ms
4aFetch live conditions from ConvexOutbound HTTP~40 ms
4bVector search against Convex vectorIndexOutbound HTTP~50 ms
4cFetch news summariesOutbound HTTP~30 ms
5Assemble + trim prompt to 2,500 tokensCPU (trivial)~1 ms
6Stream from Together.ai 70BOutbound HTTP~1–4 s
7Validate output (regex + faithfulness)CPU (trivial)~1 ms
8Stream SSE tokens to userNetwork out(concurrent with 6)
9Fire-and-forget tracking insertAsync HTTP~0 ms (non-blocking)

CPU work on the server: < 5 ms per request. The rest is waiting on network I/O. This means a very small instance handles a very large number of concurrent requests — each one is mostly idle while waiting for Together.ai to respond.

2. Recommendation: EC2 t3.small

Go with EC2 t3.small — 2 vCPU, 2 GB RAM, ~$15/mo reserved or ~$19/mo on-demand. It matches the workload, integrates with your existing AWS networking/IAM, and gives you a clear upgrade path. Lightsail is tempting but too limiting once you need VPC integration, CloudWatch, or security groups.

3. All options compared

OptionVerdictRationale
EC2 t3.small Recommended 2 vCPU, 2 GB RAM. Burstable, perfect for I/O-bound orchestration. ~$15/mo reserved. Existing AWS account with networking in place.
EC2 t3.medium If pgvector local 2 vCPU, 4 GB RAM. ~$30/mo reserved. Pick this only if you co-locate PostgreSQL + pgvector on the same box instead of using Convex's vector index.
Lightsail $10 Too limiting 2 GB RAM, 1 vCPU. Simpler networking, but you lose VPC integration, IAM roles, CloudWatch, and the ability to scale without migrating. Good for isolated projects — this isn't one.
Lambda + API Gateway No for v1 SSE streaming is awkward on Lambda (30s default timeout, response-streaming adds complexity). The long-lived Together.ai inference call (~1–4s) doesn't fit the Lambda execution model well. Revisit post-launch if you want scale-to-zero.
Fargate / ECS Overkill for v1 Adds container orchestration complexity for a single-process server. Makes sense at scale (multiple containers, auto-scaling), but for POC → soft launch you'd be over-engineering.
App Runner Decent alternative Simpler than ECS, auto-scales, ~$7/mo minimum. But less control than EC2, and you're already familiar with EC2 on this account.

4. Bare-minimum hardware spec

ResourceMinimumWhy
CPU2 vCPU (burstable)Regex classifiers, JSON assembly, SSE streaming. Bursts handle concurrent requests.
RAM2 GBNode.js/Python process + OS. No ML models running locally.
Disk20 GB gp3OS + app code + logs. The corpus and models live off-box.
NetworkStandard VPCOutbound HTTPS to Together.ai + Convex. Inbound HTTPS from Netlify/browsers.

That's a t3.small — roughly $15/mo reserved or $19/mo on-demand.

Why so small? At your projected launch traffic (~800 chat users / 5,000 turns per month), you're looking at roughly 0.1 requests per minute on average. Even the Saturday-storm peak from the stress test plan (200 concurrent users) is manageable on a t3.medium since each request is mostly waiting on Together.ai, not burning CPU.

If you co-locate pgvector

If you decide to run PostgreSQL + pgvector on the same instance instead of using Convex's vector index, bump to t3.medium (4 GB RAM). Your corpus is small (~5,000 chunks × 1024 dims = ~20 MB of vectors), so 4 GB is plenty for both the app process and PostgreSQL.

5. Convex vector index vs. self-hosted pgvector

Your rag-plan.md currently puts the vector store on Convex (good-cormorant-17). If the orchestrator moves to EC2, you have a choice about where the vector search lives.

ApproachProsCons
Keep Convex vector index Zero DB ops; already designed; built-in filtering; schema + scripts ready Adds a network hop; Convex bandwidth costs at scale; orchestrator and data on different services
pgvector on same EC2 Single box; no external vector-search latency; full SQL power for queries You run PostgreSQL (backups, upgrades); need t3.medium for RAM; re-implement the index
RDS PostgreSQL + pgvector Managed DB; automatic backups; Multi-AZ option ~$15–30/mo for db.t3.micro; adds cost but removes DB ops burden
Recommendation: keep Convex for the vector index and put just the orchestrator on EC2. It's the fastest path — your schema, indexing scripts, and search actions are already designed for Convex. Migrate to pgvector later only if Convex bandwidth becomes a real cost issue (which at 5k turns/mo it won't be).

Data flow: Browser sends POST /api/rag/chat to the EC2 instance. The orchestrator calls Together.ai for classification and inference, Convex for vector search and live conditions, and the news API for article summaries. Tokens stream back to the browser via SSE. Tracking is fire-and-forget to Convex.

8. When to scale up

TrafficInstanceAction
< 30 req/mint3.smallNo change needed. Handles launch traffic + 10× headroom.
30–100 req/mint3.medium or t3.largeResize in place. Consider adding a second instance behind an ALB.
> 100 req/min sustainedECS/FargateContainerize and auto-scale. Or evaluate the dedicated-GPU path from the deployment plan.

Traffic context

At projected launch traffic (~800 chat users, 5k turns/mo):

  • Average: ~0.1 req/min — the server is idle most of the time
  • Saturday storm peak (stress test scenario 2): 200 concurrent users for 90 seconds — manageable on t3.medium since each request is mostly waiting on Together.ai
  • Sustained growth target (250k turns/mo): ~6 req/min average, ~60 req/min peak — still within t3.medium territory

The t3 burstable instances are ideal for this usage pattern: mostly idle with occasional bursts of concurrent requests. You earn CPU credits during idle periods and spend them during peaks.

9. Setup checklist

9.1 EC2 instance

  • Launch t3.small in your existing VPC
  • Amazon Linux 2023 or Ubuntu 24.04 LTS
  • 20 GB gp3 root volume
  • Security group: inbound 443 (HTTPS) + 22 (SSH from your IP only)
  • IAM role with CloudWatch Logs permissions
  • Elastic IP (static address for DNS)

9.2 Software stack

  • nginx — reverse proxy, TLS termination (Let's Encrypt via certbot)
  • Node.js 20 LTS — runtime for the orchestrator
  • PM2 — process manager with auto-restart and log rotation
  • certbot — auto-renewing TLS certificates

9.3 Environment variables

TOGETHER_API_KEY=tok_...
CONVEX_URL=https://good-cormorant-17.convex.site
NEWS_API_URL=https://www.snow-country.com/resorts/api-easy-blog-list.php
NODE_ENV=production
PORT=3000
RATE_LIMIT_PER_IP=30
RATE_LIMIT_WINDOW_MIN=5

9.4 DNS

  • Point api-chat.snocountry.com (or similar) A record to the Elastic IP
  • Or use the existing domain with a /api/rag/ path proxied from Netlify to the EC2 instance

9.5 Monitoring

  • CloudWatch agent — CPU, memory, disk metrics
  • CloudWatch alarms — CPU > 80% sustained, disk > 85%, status check failed
  • PM2 logs → CloudWatch Logs for centralized log search
  • Application-level — the rag_requests table on Convex handles per-request metrics (tokens, cost, latency, errors)

9.6 Backups & disaster recovery

  • The EC2 instance is stateless — all state lives in Convex (vector index, tracking) and Together.ai (model weights). A replacement instance can be provisioned in minutes.
  • Keep the deployment script in your repo for reproducibility
  • Weekly AMI snapshot for faster recovery (optional but cheap)

9.7 Security hardening

  • No inbound ports except 443 + 22 (SSH restricted to your IP)
  • TOGETHER_API_KEY in AWS Systems Manager Parameter Store (encrypted), not in env files on disk
  • Auto-update OS security patches (Amazon Linux: dnf-automatic)
  • fail2ban for SSH brute-force protection
Bottom line: An EC2 t3.small at ~$15–19/mo is the right starting point. The RAG server is an orchestrator — the expensive compute (LLM inference) stays on Together.ai. Start small, monitor real traffic, and resize in place when the data tells you to.
