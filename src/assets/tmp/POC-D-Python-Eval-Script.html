Complete reference for the Python eval script that tests Llama 3.1 8B, Qwen 3.5 9B, and Llama 3.3 70B against 18 questions. Zero external dependencies — uses only Python's standard library (urllib, json, csv, re). See Eval Script Design Philosophy for the reasoning behind each design decision and Model Evaluation Question Set for the 18 questions and scoring rubric.

2. File roles at a glance

FileTypePurposeWhen you touch it
run_eval.py Orchestrator Main entry point. Runs all 3 phases in sequence: runner → scorer → report. Also supports --score-only and --report-only for re-processing existing results. Rarely — it's the stable glue layer
config.py Configuration All tunable parameters in one place: model IDs, Together.ai pricing, temperature, max tokens, directory paths, faithfulness ignore list. When you add/swap models, change token limits, or update pricing
runner.py Phase 1 Assembles prompts from fixtures, calls Together.ai API for each model × question, saves raw JSON responses with metadata (tokens, latency, cost). Only if you change the prompt template format
scorer.py Phase 2 Mechanical scoring: extracts numbers from responses and verifies against fixture values (faithfulness), detects pass/fail signal phrases (guardrails), generates CSV template for human scoring. When you add new scoring rules or adjust the ignore list
report.py Phase 3 Generates a self-contained HTML comparison report with side-by-side model responses, auto-scores, latency, cost, and flagged issues per question. When you want to customize the HTML output format
fixtures/questions.json Fixture The 18 eval questions with full RAG context blocks (LIVE_DATA, KNOWLEDGE, NEWS), expected behavior descriptions, faithfulness values, and pass/fail signal lists. When you add, edit, or refine questions or context data
fixtures/system_prompt.txt Fixture The hardened system prompt sent identically to all 3 models. Matches the production prompt from rag-plan.md. When you update the production system prompt

Each phase reads the output of the previous phase. Phases 2 and 3 can be re-run independently on existing results:

# Re-score without re-running API calls
python run_eval.py --score-only results/eval-20260608-201500.json

# Re-generate report without re-scoring
python run_eval.py --report-only results/eval-20260608-201500.json
Phase 1 costs money (~$0.50). Phases 2 and 3 are free — they only process local JSON files. You can re-score and re-generate reports as many times as you want without additional API calls.

4. run_eval.py — main entry point

PY
run_eval.py
Orchestrator — runs all 3 phases in sequence
View

What it does

The single command you run. It imports runner, scorer, and report, then executes them in order. Handles argument parsing for all operational modes (full run, dry run, single model, single question, score-only, report-only).

Key functions

  • main()
    Parses CLI arguments, dispatches to the correct mode, prints final summary with next-steps checklist.
  • find_latest_results()
    Finds the most recent eval-TIMESTAMP.json in results/ (excludes scored/comparison files).
  • run_scorer(path, verbose)
    Wrapper that loads results JSON, scores each result, prints summary, generates CSV and scored JSON.
  • run_report(path)
    Wrapper that loads results JSON and generates the comparison HTML.

CLI flags

FlagExampleWhat it does
--dry-runpython run_eval.py --dry-runShows prompt sizes and model IDs without calling the API. Free.
--model Npython run_eval.py --model 0Run only one model (0=Llama 8B, 1=Qwen 9B, 2=Llama 70B).
--question IDpython run_eval.py --question Q07Run only one question across all models.
--verbose / -vpython run_eval.py -vShow detailed per-question scoring in terminal output.
--score-only FILEpython run_eval.py --score-only results/eval-*.jsonSkip API calls. Re-run scorer on existing results.
--report-only FILEpython run_eval.py --report-only results/eval-*.jsonSkip API calls + scoring. Re-generate HTML report only.

6. runner.py — API caller & prompt assembler

PY
runner.py
Phase 1 — calls Together.ai for each model × question
View

What it does

Loads the system prompt and questions from fixtures/, assembles prompts using XML-fenced RAG context blocks (matching the production prompt template from rag-plan.md), calls the Together.ai chat completions API, and saves both individual raw JSON files and a combined results file.

Key functions

  • load_system_prompt()
    Reads fixtures/system_prompt.txt and returns it as a string.
  • load_questions()
    Reads fixtures/questions.json and returns the parsed data including categories and all 18 questions.
  • build_user_message(question_obj)
    Assembles the user-role message from the question's context fixture. Wraps live data in <LIVE_DATA>, knowledge chunks in <KNOWLEDGE>, and news in <NEWS> tags with untrusted="true" attributes — same XML fencing the production RAG pipeline uses.
  • call_together(model_id, system_prompt, user_message)
    Makes the HTTP POST to Together.ai using urllib (no requests dependency). Returns a dict with content, token counts, latency, finish reason, or error details. Timeout: 30 seconds.
  • calc_cost(model_cfg, input_tokens, output_tokens)
    Calculates USD cost for a single request using the per-million-token rates from config.py.
  • run_eval(model_filter, question_filter, dry_run)
    Main loop. Iterates all questions × models, calls the API, saves results. Prints progress like [14/54] Q05 × Qwen 3.5 9B ... OK (820ms, 412+187 tok, $0.0001). Adds 200ms courtesy delay between calls.

Prompt template

The assembled user message follows this structure (matching production):

<LIVE_DATA untrusted="true">
[{"resort": "Vail", "baseDepth": 54, ...}]
</LIVE_DATA>

<KNOWLEDGE untrusted="true">
[1] Wolf Creek is known as Colorado's hidden gem...
</KNOWLEDGE>

User question: What are current conditions at Vail?

Output

Saves to results/:

  • results/raw/TIMESTAMP_Q01_Llama-3.1-8B.json — one file per call (54 total)
  • results/eval-TIMESTAMP.json — combined file with all 54 results + summary

7. scorer.py — mechanical faithfulness & signal checks

PY
scorer.py
Phase 2 — auto-scores every response mechanically
View

What it does

Runs three types of automated checks on each model response, generates a CSV template pre-filled with auto-scores (with blank columns for human scoring), and prints a summary with flagged issues.

Key functions

  • extract_numbers(text)
    Regex-extracts all numeric values from a response. Returns a set of string representations (e.g. {"54", "121", "6"}).
  • check_faithfulness(response_text, allowed_values, ignore_set)
    The core faithfulness checker. Extracts numbers from the response, compares each against the faithfulness_values set from the question fixture, and classifies as verified, unverified, or ignored. Returns a ratio (verified / total checked). A ratio below 1.0 means the model cited a number not in the RAG context — a potential hallucination.
  • check_signals(response_text, fail_signals, pass_signals)
    Case-insensitive search for defined signal phrases. Returns FAIL if any fail signal is found, PASS if pass signals are found (and no fail signals), REVIEW if neither match clearly, or N/A if no signals were defined for that question.
  • score_result(result)
    Scores a single model × question result. Runs faithfulness + signal checks, computes word count, and returns a structured score object.
  • generate_csv(scores, output_path)
    Writes a CSV with auto-filled columns (faithfulness ratio, signal status, word count, latency, cost, response preview) and blank columns for human scoring (faithfulness 1–5, relevance 1–5, tone 1–5, guardrail pass/fail, notes).
  • print_summary(scores, verbose)
    Prints per-model aggregates (avg word count, avg latency, faithfulness %, signal pass/fail counts, total cost) and a list of flagged issues (unverified numbers, fail signals triggered).

Faithfulness example

ScenarioResponse excerptAllowed valuesResult
Clean pass"Vail has 54-inch base with 121 trails open"[54, 121, 195, ...]100% — all numbers verified
Hallucination caught"Vail has 54-inch base with 150 trails open"[54, 121, 195, ...]50% — "150" is unverified
Ignored numbers"In 2026, Vail has 54-inch base"[54, 121, ...]100% — "2026" is in ignore set

CSV template output

Designed to be opened in Google Sheets. Auto-filled columns let you see the automated analysis; you fill in the human scores by reading each response in the comparison HTML report.

Auto-filledHuman fills
question_id, category, model, word_count, faithfulness_ratio, faithfulness_unverified, signal_status, latency_ms, cost_usd, response_previewfaithfulness_human (1–5), relevance_human (1–5), tone_human (1–5), guardrail_human (pass/fail), notes

8. report.py — HTML comparison generator

PY
report.py
Phase 3 — visual side-by-side comparison report
View

What it does

Reads the combined results JSON and generates a self-contained HTML file with all 18 questions displayed side-by-side across the 3 models. Each model response card shows the full text, auto-scores (faithfulness, signals), latency, cost, and flagged issues. The report is designed for stakeholder review — shareable and readable without technical context.

Key functions

  • build_html(results, summary)
    The main generator. Groups results by question, renders them in Q01–Q18 order with category badges, expected behavior descriptions, and 3-column model cards. Calls scorer.check_faithfulness() and scorer.check_signals() inline to compute auto-scores for the visual display. Returns a complete HTML string with embedded CSS.

Report layout

  • Header bar — timestamp, question × model count, total cost
  • Summary cards — per-model cost and average latency
  • Question sections — one per Q01–Q18, each with:
    • Category badge with weight percentage (e.g. Narrative & trip planning 25%)
    • Expected behavior description and constraints
    • 3 model response cards in a responsive grid with:
      • Full response text (scrollable at 300px max height)
      • Word count, latency, cost, token usage
      • Faithfulness percentage with color coding (green ≥ 90%, amber ≥ 70%, red < 70%)
      • Signal status badge (PASS / FAIL / REVIEW)
      • Unverified numbers warning if any hallucinations detected

Output

results/eval-TIMESTAMP-comparison.html — open in any browser. No server needed.

9. Fixtures — the controlled inputs

FX
fixtures/questions.json
18 questions + RAG context + expected behavior
View

Structure

Top-level object with categories (C1–C7 with names and weights) and a questions array. Each question object contains:

FieldTypePurpose
idstringStable identifier ("Q01"–"Q18")
categorystringCategory key ("C1"–"C7")
questionstringThe user's question — exactly as typed
context.live_dataarrayMock resort data objects (what Convex would return)
context.knowledge_chunksarrayMock vector-retrieved prose paragraphs
context.newsarrayMock news article summaries
faithfulness_valuesnumber[]Every number the model is allowed to cite. Used by scorer.py.
expected_behaviorstringHuman-readable description of what a good answer looks like
fail_signalsstring[]Phrases that indicate failure (e.g. "Paris" for injection Q17)
pass_signalsstring[]Phrases that indicate correct handling (e.g. "snow conditions")
constraintsstring[]Named constraints for trip-planning questions (for display in report)
tagsstring[]Filterable labels (future use)

How to add a question

  1. Pick the next ID ("Q19")
  2. Assign a category ("C1"–"C7")
  3. Write the question as a real user would phrase it
  4. Build the context fixture — what data would the RAG pipeline retrieve for this question?
  5. List every number in the fixture as faithfulness_values
  6. Define fail_signals / pass_signals for guardrail and edge-case questions
FX
fixtures/system_prompt.txt
Hardened system prompt — identical for all models
View

Plain text file. Contains the 8 rules from the production system prompt: faithfulness mandate, graceful degradation, topic boundaries, prompt concealment, conciseness target, tone guidance, data-driven comparisons, and multi-constraint acknowledgment.

Keep this in sync with your production prompt. If you update the production system prompt in rag-plan.md or the Convex action, update this file to match.

10. Output files — what gets generated

All output goes to eval/results/. Each run is timestamped so multiple runs don't overwrite each other.

FileCreated byFormatWhat's in itWhat you do with it
results/raw/*.json runner.py JSON (54 files) One file per API call. Full request/response with system prompt length, user message, model output, token counts, latency, cost, plus the question's fixture data for self-contained debugging. Deep-dive into a specific model × question. Rarely opened manually.
results/eval-TIMESTAMP.json runner.py JSON Combined results from all 54 calls. Includes config metadata and cost summary. This is the canonical artifact — everything downstream reads from it. Input to scorer.py and report.py. Archive for reproducibility.
results/eval-TIMESTAMP-scored.json scorer.py JSON Auto-score results: faithfulness ratio, verified/unverified numbers, signal status per model × question. Programmatic access to scores. Can be imported into notebooks or dashboards.
results/eval-TIMESTAMP-scores.csv scorer.py CSV Spreadsheet template with auto-filled columns + blank human-score columns (faithfulness 1–5, relevance 1–5, tone 1–5, notes). Open in Google Sheets. Fill in human scores while reading the comparison HTML. This is your scoring worksheet.
results/eval-TIMESTAMP-comparison.html report.py HTML Visual side-by-side comparison of all 3 models on all 18 questions. Category badges, auto-scores, flagged issues, full response text. Open in browser. The primary review artifact. Read each response, then fill in human scores in the CSV.

11. Quick-start commands

First-time setup

# No pip install needed — zero external dependencies
cd eval/
export TOGETHER_API_KEY=tok_your_key_here

Full eval run (~$0.50, ~2 min)

python3 run_eval.py

Dry run (free — preview prompts)

python3 run_eval.py --dry-run

Test a single model first

# Test cheapest model first to verify API key works
python3 run_eval.py --model 0 --question Q01

Test a single question across all models

python3 run_eval.py --question Q07

Re-score existing results (free)

python3 run_eval.py --score-only results/eval-20260608-201500.json

Re-generate report (free)

python3 run_eval.py --report-only results/eval-20260608-201500.json

Verbose scoring (see per-question detail)

python3 run_eval.py -v

Run scorer directly

python3 scorer.py results/eval-20260608-201500.json --verbose

Run report generator directly

python3 report.py results/eval-20260608-201500.json
Recommended first run: Start with python3 run_eval.py --model 0 --question Q01 to verify your API key works and see the output format. Then run the full eval with python3 run_eval.py. Total cost for the full run: ~$0.50.
